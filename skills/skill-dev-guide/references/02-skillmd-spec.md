# SKILL.md 写作规范

{{ASSISTANT_NAME}}体系 Skill 的 SKILL.md 文件格式与字段约束。

## 行数限制

- **≤ 500 行**
- 超出则拆分到 `references/`

## Frontmatter(YAML)

```yaml
---
name: skill-name
author: 杜重阳律师（微信Dcylawer8888)
version: "1.0.0"
license: MIT
description: |
  本技能应在用户需要[场景]时使用。
  不要用于：[负向条件1]、[负向条件2]。
---
```

## 字段说明

| 字段 | 必须 | 说明 |
|------|------|------|
| `name` | ✅ | 技能标识 |
| `author` | ✅ | 作者,固定为 "{{USER_FULL_NAME}}（微信{{USER_WECHAT}})" |
| `version` | ✅ | SEMVER 版本号,与 CHANGELOG.md 最新版本一致 |
| `license` | ✅ | MIT / CC-BY-NC / 其他 |
| `description` | ✅ | 含触发条件 + 负向条件,≤ 1024 字符 |

## description 写作原则：路标，不是说明书

**Skill 描述的唯一目的：让 Agent 在路由时一眼认出这个 skill。仅此而已。** 描述越长，Agent 的噪声越多，选错概率越高。重逻辑、详细步骤、特殊情况全部放进正文或 `references/`——不要把整本说明书挂在路标上。

核心约束：

- **≤ 5 行 / ≤ 200 字符**（含触发+负向条件）
- 使用第三人称:"本技能应在..."
- 必须包含**负向触发条件**:"不要用于:..."
- 1-3 条负向条件即可
- 写完自检：删到不能再删，每多一个词都是在跟其他 skill 抢注意力

> 经验数据：描述从 90+ 词砍到 40 词以内，Agent 选对率显著提升。中文同理。

**反例**（太啰嗦）：
```yaml
description: |
  本技能应在用户收到法院送达的各类诉讼文书（包括但不限于裁判文书、裁定书、判决书、
  调解书、应诉通知书、举证通知书等）时使用，自动解析案号、当事人信息、文书类型、
  送达日期等关键信息，并完成文书下载、归档、待办提醒创建等一系列操作。
  不要用于：非法院来源的普通文件处理、合同审查、法律咨询。
```

**正例**（路标）：
```yaml
description: |
  本技能应在用户收到法院12368短信（文书送达、开庭提醒等）时使用，
  自动提取案号并完成文书下载归档+待办创建。
  不要用于：合同审查、法律咨询、普通文件处理。
```

## 结构模板

```markdown
---
frontmatter
---

# 技能名称

## 功能概述
一句话说清这个技能做什么

## 调用方式
触发条件、调用入口

## 工作流
分步骤描述

## 与其他技能配合
引用的上下游 Skill

## 配置
外部依赖说明(如有)

## 变更历史
见 CHANGELOG.md
```

## 写作风格

- 简洁直接,不堆砌客套话
- 用 markdown 标题分章节,不用过多嵌套
- 关键约束用表格或列表,易扫读
- 引用其它 skill 时只引不抄
