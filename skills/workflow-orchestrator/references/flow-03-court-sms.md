## 流程三：法院短信接收与文书处理

适用场景：{{USER_SHORT_NAME}}收到法院短信（文书送达、立案通知、开庭提醒等），粘贴短信内容或送达链接。

### 执行方式

本流程委托 [[court-sms]] 技能完整执行。[[court-sms]] 内部含六步工作流：

1. **输入解析** — 识别短信类型、提取案号/当事人/下载链接
2. **确定归档目录** — 案件管理 `{当事人}/` 目录
3. **文书下载** — 调用对应送达平台 API 下载文书
4. **归档保存** — 文书归档 + 上诉期限计算 + 结构化汇报

> 详细执行步骤见 `court-sms/SKILL.md`。本文件仅定义流程三与 orchestrator 主流程的衔接规则。

### 前置条件

- 无。[[court-sms]] 自行处理文书 PDF 解析(走 [[markitdown]] + 自有 PDF 解析逻辑)。
- 如 court-sms 内部流程需要用户输入（如验证码、手动选择案件），就地处理，不回传 orchestrator。

---

### court-sms 完成后的衔接（关键）

court-sms 第四步汇报完成后，**不直接结束**。orchestrator 根据下载的文书类型，主动衔接后续法律工作：

| 下载的文书类型 | 衔接动作 | 路由目标 |
|-------------|---------|---------|
| **判决书 / 裁定书** | 询问："是否需要我对该判决/裁定进行分析？（上诉策略/履行评估/申请执行）" | → 流程一（[[markitdown]]→[[process-cases]]） |
| **传票（开庭通知）** | 询问："是否需要我协助庭前准备？（证据梳理/攻防演练/庭审模拟）" | → [[legal-evidence-mapping-mctmilk]] 或 [[legal-debate-simulation-mctmilk]] |
| **起诉状 / 上诉状（对方）** | 询问："是否需要我分析对方主张并准备答辩策略？" | → 流程一（[[markitdown]]→[[process-cases]]） |
| **举证通知书** | 询问："是否需要我按举证通知梳理证据清单？" | → [[legal-evidence-mapping-mctmilk]] |
| **其他文书** | 询问："是否需要我对该文书进行法律分析？" | → analyze-legal-issues |
| **多种文书混合** | 逐项列出，让{{USER_SHORT_NAME}}选择优先处理哪份 | → 按选择路由 |

**兜底**：无论何种文书类型，归档完成后**必须**询问："是否需要更新案件进展日志？" → [[case-progress-archive]]

### 衔接对话示例

```
[ court-sms 完成汇报 ]

📋 已归档 3 份文书至 `待处理/张三/`：
- 民事判决书（2025）苏0981民初1234号
- 开庭传票 — 2026年6月15日 09:30 第3法庭
- 举证通知书

⏰ 上诉截止日期：2026年6月10日（判决书送达后15天）

---

检测到判决书和传票。是否需要我：
1. 分析该判决，评估上诉策略？
2. 协助庭前准备（证据梳理+攻防演练）？
3. 先更新案件进展日志？
```

### 边界

- court-sms 内部文书 PDF 文字提取使用其自有的 PDF 解析逻辑,不经过 [[format-conversion]] 的统一策略(2026-06-05 重构前为 `ocr-strategy.md`,已退役)。
- court-sms 不处理合同审核——合同类文件走 [[contract-review-pro]]。
- 如 court-sms 下载失败（网络/验证码/平台限制），在汇报中说明后正常结束，不触发衔接询问。
