## 流程一：案件全流程处理（起草→归档标准线）

### 步骤 0：MarkItDown / LLM 多模态预处理(原 OCR 预处理,2026-06-05 重构)

律师提供材料后,先进行格式转换。调用策略详见 [[format-conversion]](已退役的 OCR 策略见 `references/ocr-strategy.md.bak` 或 CHANGELOG 历史)。

**关键行为：**
- 自动识别材料类型(图片/文档/混合),按策略分发到对应处理路径
  - 文档(PDF/Office) → MarkItDown.convert_local()
  - 图片 → 当前 LLM(MiniMax-M3)vision 接口
  - 扫描件 PDF → MarkItDown 失败时降级 LLM vision 逐页读图
  - 音频/HTML/CSV/JSON/XML/EPUB/YouTube → MarkItDown
- 转换完成后,将 Markdown 文本作为后续分析的材料内容
- 如有文件转换失败,在步骤 1 的 Hooks 中一并汇报

**软 Hooks：** 转换完成后,向律师汇报:
- 共接收 N 个文件,成功转换 M 个
- 如有失败文件,列出清单并提示可提供可读版本

### 步骤 1：接收材料→分析

基于 Markdown 输出文本,进行分析(调 [[process-cases]]),给出案件简要。

> ⚠️ **执行顺序说明**：调用 [[process-cases]] 时，仅取其**分析输出**（案情摘要、法律问题分析），不执行其内部的完整交互流程。分析输出后，回到本流程的硬 Hooks 节点等待律师确认。

**硬 Hooks：** 让律师确认以下内容后方可继续——
- 我方诉讼地位（原告/被告/第三人）
- 材料内容简要是否准确
- 是否有其他需要补充的重点
- 律师目前的诉讼思路
- 需要起草的材料类型（诉状、辩论意见、案件提纲、证据分析等）

### 步骤 2：策略确认

得到律师反馈后，进一步分析。如有律师没有回复的问题，执行软 Hooks（提醒律师对未回复问题进行回复）；如无则给出案件分析建议，并附法律依据且核验法条有效性。

策略分析应包含以下要素：
- **法律关系定性**：本案涉及何种法律关系
- **请求权基础**：原告/我方主张的法律依据
- **举证责任分配**：各方需证明的事实
- **风险识别**：信息缺失点、逻辑漏洞、对方可能的抗辩路径
- **建议策略**：首选方案及备选方案

**硬 Hooks：** 请求律师确认诉讼策略。

**软 Hooks 规则：** 就步骤 1 中律师未回复的问题提醒一次，仍未回复则跳过，不阻塞流程。

### 步骤 3：起草文件

根据确认的策略，按律师指示起草。如律师未明确指定起草类型，根据分析结果推荐应起草的文书清单供律师选择。

常见文书类型包括但不限于：
- 起诉状 / 上诉状 / 反诉状
- 答辩状
- 代理词 / 代理意见
- 辩论意见 / 辩论提纲
- 质证意见
- 证据目录 / 证据清单
- 保全申请书
- 管辖权异议申请书
- 追加当事人申请书
- 调查取证申请书
- 撤诉申请书
- 延期审理申请书

**硬 Hooks：** 起草前让律师确认文书类型和核心要点。

### 步骤 4：初稿展示与修改

获得确认后，在对话中直接展示初稿正文供审阅（不用 MD 文件预览）。

由律师在对话中确认或提出修改意见：
- 律师确认 → 进入步骤 5
- 律师提出修改意见 → 修改后重新展示，再次确认
- 修改循环不超过 3 轮；第 3 轮后如仍未定稿，建议律师面谈确认关键争议点

### 步骤 5：定稿

律师确认初稿后，转 DOCX（调 [[draft-legal-docs]] 步骤 3 → 委托 [[word-docx]] 处理具体转换），并直接进入步骤 6。

### 步骤 6：归档

调 [[case-progress-archive]]，将定稿文件及案件进展日志上传至 `\金山文档云盘\案件管理\` 下的对应客户名称文件夹。

**归档路径规则：**
- 在 `案件管理\` 下查找匹配当前案件的客户名称文件夹
- 如找到 → 直接归档至该文件夹
- 如未找到 → **硬 Hooks**：请{{USER_SHORT_NAME}}告知该案对应的文件夹命名（客户名称），确认后归档
